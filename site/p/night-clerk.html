Pitch
Discovery should live where the need occurs: the terminal. A bundled catalog
snapshot makes the marketplace self-recommending, offline, with zero invention.
Spec
- One skill, clerk, description (verbatim): "Search the Nightshift Foundry
  catalog and recommend plugins for a task. Use when the user asks what plugin
  helps with X, browse the foundry, or find me a plugin."
- Reads bundled data/catalog.json (published entries only: name, one_liner, tags,
  install line, snapshot date), generated by tools/clerkcat.py; the maintainer
  regenerates + patch-bumps the clerk whenever the shelf changes.
- Answers with <=3 fits + exact install lines + the snapshot date; if nothing
  fits, says so and links the idea template. NEVER invents a plugin or capability.
Experiment
- Hypothesis: the clerk becomes the top referrer of installs among plugins
- Metric: installs proxy (unique clones) after clerk release vs before; Baseline: 0; Review-after: 2026-09-05

### Review — i59 (reviewer)
- A marketplace that recommends itself could so easily lie; this one structurally
  can't — the skill may only speak from a generated, published-only catalog, and
  the suite proves the subset relation every run.
- Snapshot-date disclosure in every answer is the honest fix for bundled data;
  the anti-lore clause keeps the clerk a clerk, not a mascot.
- Sharpest question: once published, the clerk appears in its own catalog — is
  self-recommendation a conflict? No: it competes on the same one_liner as
  everyone else, and "the front desk recommends the front desk" only fires when
  the user literally asks for discovery help.
