Discovery-only audit. Every item cites repo evidence; nothing here is implemented.
The previous slate (post-v8, i136/ADR-016) was built in full as v9 — 13/13 lawful
per `reviews/audit-004.md` — and is preserved in git history. This slate is deduped
against it: nothing below repeats a v9 item.

## 1 · Product snapshot

Nightshift Foundry is a self-running Claude Code plugin workshop that is also the
marketplace it publishes to (`.claude-plugin/marketplace.json`) and the storefront
that displays it (`site/`, generated). 151 iterations in: 38 records (32 published),
8 installable plugins, 4 self-audits, executable QA suites (`tools/qa.sh`), weekly
re-verification (ADR-013, `tools/restamp.py`), a budget governor, release tags, and
a go-live preflight (`tools/preflight.py`) reporting 7 OK / 3 operator-optional.
The machinery is complete; the audience is zero — `state/METRICS.jsonl` shows
0 stars and null traffic, and ROADMAP Gate A waits only on the operator's 15
minutes. This slate therefore leans away from more spectacle and toward four lanes
the machine can work while the window is dark: end-user value on the shelf itself,
the untested parts of its own gates, productization gaps that bite at first contact,
and two differentiators no other repo can copy.

## 3 · Top 3 (sketches in §4 under each item)

1. **`gates.yml`** (#5) — the biggest CI hole: a PR touching `foundry/records/`,
   `state/`, or `tools/` currently merges with zero checks.
2. **Tests for the gate itself** (#6, folds in #8) — `tools/build.py` is 1,311
   lines, `tools/validate.py` 354, coverage zero; every other guarantee stands on
   them.
3. **Verified-by-Foundry** (#13) — the repo already owns the ecosystem's only
   verification machinery; packaging it makes the foundry the trust layer for all
   Claude Code plugins, not just its own shelf.

## 4 · Full list

### Lane A — end-user value

#### 1 · Update awareness for installers — IMPROVE
- **Evidence:** the Version law (`LOOP.md` §3) guarantees every change to a
  published plugin bumps semver + CHANGELOG, but no surface delivers that fact to
  installed users: `plugins/night-clerk/data/catalog.json` (generated by
  `tools/clerkcat.py`) carries no `version` field, and nothing in any plugin tells
  a user a new release exists.
- **Proposal:** add `version` to the clerk's snapshot schema, and give night-clerk
  a `whats-new` skill that compares installed plugin versions against the snapshot
  (version law: clerk bump + CHANGELOG + tag).
- **Why:** the law exists so installed users receive updates; today they receive
  them silently or never notice. **Impact 4 · Effort 2.**

#### 2 · Hook configurability convention — IMPROVE
- **Evidence:** `plugins/test-gap-nudge/scripts/nudge.sh` hardcodes its
  source-extension list (`SRC_EXT='py|js|ts|…'`); commit-craft's message-guard
  rules are equally fixed. The hooks-are-guests law covers safety but offers users
  no knobs.
- **Proposal:** a documented env-var convention (`TEST_GAP_NUDGE_EXTS=…`,
  `COMMIT_CRAFT_TYPES=…`) applied across hook plugins, each with a README section
  and a qa-suite check that the default path still fails open.
- **Why:** the alternative to tuning a guest is uninstalling it. **Impact 3 ·
  Effort 2.**

#### 3 · The clerk on the storefront — NEW
- **Evidence:** night-clerk's recommendation logic only helps people who already
  installed night-clerk; `site/index.html` (543 lines) already loads
  `site/data.json` with every plugin's tags, one-liners, and install lines.
- **Proposal:** a small static picker on the window ("what are you working on?" →
  kit or plugin + copyable install block), pure client-side over data.json —
  template change, so ADR first (two-iteration rule).
- **Why:** gives visitors the clerk's guidance at the moment of discovery, feeding
  the installs-per-visitor experiment already armed. **Impact 3 · Effort 2.**

#### 4 · Rendered plugin READMEs on certificate pages — IMPROVE
- **Evidence:** `site/p/*.html` provenance pages tell each plugin's build story,
  but a visitor cannot read what the plugin actually *does* without leaving for
  the GitHub file tree — the shipped `plugins/<name>/README.md` is rendered
  nowhere on the window.
- **Proposal:** `tools/build.py` embeds the shipped README (fenced, it is already
  trusted repo content) into each certificate page under the provenance log.
- **Why:** the storefront should be self-contained at the moment of the install
  decision. **Impact 3 · Effort 2.**

### Lane B — developer experience

#### 5 · `gates.yml`: validate+build on every PR — FIX
- **Evidence:** `.github/workflows/qa.yml` is path-filtered to `plugins/**`,
  `foundry/tests/**`, `tools/qa.sh`; the only place `validate.py`/`build.py` run
  in CI is `deploy-site.yml`, which triggers on push to main only. A PR touching
  `foundry/records/`, `state/`, or `tools/` merges with zero checks — mode:pr
  shifts (ADR-008/017) are exactly such PRs.
- **Proposal:** a ~20-line `gates.yml` on `pull_request` + `push` (all paths):
  `python3 tools/validate.py && python3 tools/build.py`, then
  `git diff --exit-code site/ foundry/INDEX.md` to prove generated output was
  committed in sync. Workflows sit adjacent to `tools/`, so file the ADR alongside.
- **Why:** the human-veto lane reviews PRs whose gates never ran. **Impact 4 ·
  Effort 1.**

#### 6 · Pytest suite for the gates themselves — IMPROVE
- **Evidence:** `tools/build.py` (1,311 lines) and `tools/validate.py` (354) have
  zero test coverage — `tools/qa.sh` only discovers `foundry/tests/<plugin>/`
  suites. The tools that enforce every law are the only untested code in the repo.
- **Proposal:** `foundry/tests/_tools/` (or `tools/tests/`) with fixture repos in
  tmp dirs: a valid record/plugin pair plus mutations (missing stage section, bad
  semver, marketplace desync, hook with `.*` matcher, unterminated front matter)
  asserting each specific validator error fires; wire into gates.yml. Two-iteration
  ADR applies (`tools/`).
- **Why:** a regression in the gate is invisible precisely because everything
  downstream trusts it. **Impact 4 · Effort 3.**

#### 7 · qa.sh silent-skip fix — FIX (already filed P3, audit-004 / i151)
- **Evidence:** `tools/qa.sh` `run_suite`: a suite dir containing no *executable*
  `*.test.sh` prints a note and passes — one lost chmod bit silently disables a
  published plugin's entire suite while "TEST VERDICT: pass" keeps its green check.
- **Proposal:** when the record's stage is rc+ (readable from front matter), a
  suite dir with zero executable tests counts as a fail; pre-rc keeps the current
  grace.
- **Why:** the tripwire law exists because quiet passes are how inspection goes
  soft. **Impact 3 · Effort 1.**

#### 8 · Deduplicate `parse_front_matter` — IMPROVE
- **Evidence:** the hand-rolled front-matter parser exists verbatim in both
  `tools/validate.py:45` and `tools/build.py` — the two gates could drift into
  parsing the same record differently.
- **Proposal:** extract into `tools/lib.py` with the stage/section tables; both
  gates import it. Natural first step of #6 (the fixtures test the shared lib).
- **Why:** one parser, one truth. **Impact 2 · Effort 1.**

### Lane C — productization

#### 9 · Root SECURITY.md + private vulnerability channel — FIX
- **Evidence:** `charter/SECURITY.md` is internal law; `.github/ISSUE_TEMPLATE/`
  offers bug/idea/question/field-report — all public. These plugins run hooks on
  strangers' machines, and there is no GitHub-recognized security policy or
  private reporting path.
- **Proposal:** root `SECURITY.md` (GitHub surfaces it on the repo and in new-issue
  flows): scope, supported versions (published shelf), private-report instructions
  (GitHub private vulnerability reporting), SLO pointing at the bug lane that
  already outranks new builds (LOOP.md priority 3).
- **Why:** table stakes for software that executes on install. **Impact 3 ·
  Effort 1.**

#### 10 · Opt-in debug mode for fail-open hooks — IMPROVE
- **Evidence:** `plugins/test-gap-nudge/scripts/nudge.sh` opens with
  `exec 2>/dev/null` and `trap 'exit 0' ERR` — correct per hook law, but it means
  field breakage is invisible *by design*; a user whose nudge never fires has no
  way to see why.
- **Proposal:** a documented `<PLUGIN>_DEBUG=1` convention: when set, hooks log
  their decision trail to `${TMPDIR}/…` instead of the void; README section +
  qa-suite check that debug-off behavior is byte-identical.
- **Why:** turns "fails silently" into "diagnosable" — and feeds the organic-bug
  lane Gate B is waiting on. **Impact 3 · Effort 2.**

#### 11 · Pin the CI toolchain — FIX
- **Evidence:** `qa.yml` runs `npm install -g @anthropic-ai/claude-code` (floating
  latest) in both the per-push job and the weekly reverify — so a red weekly run
  may mean "upstream CLI changed," not "plugin regressed," and `tested_with`
  stamps float with it.
- **Proposal:** pin the version in one place (an Actions variable or a versions
  file), bump deliberately via the existing `dep-bump-brief` record's lane, and
  let reverify report the pinned version in its stamps.
- **Why:** a re-verification whose harness drifts isn't verifying the same thing
  twice. **Impact 3 · Effort 1.**

#### 12 · Versioned marketplace snapshot on releases — IMPROVE
- **Evidence:** `release-on-tag.yml` lays annotated `<name>-v<version>` tags (the
  release ledger), but a release contains no artifact — an installer who wants a
  known-good state has nothing to pin or roll back to except git archaeology.
- **Proposal:** on tag, attach a zip of `plugins/<name>/` + the marketplace entry
  as release assets.
- **Why:** rollback becomes an operation instead of an excavation. **Impact 2 ·
  Effort 2.**

### Lane D — differentiation

#### 13 · "Verified by the Foundry" — a trust registry for third-party plugins — NEW
- **Evidence:** the repo already owns the ecosystem's only verification machinery:
  weekly re-verification (`tools/restamp.py`, ADR-013), token-cost measurement
  (`tools/tokencost.py`), a shields endpoint (`site/badge.json`), and
  plugin-smith's `doctor` skill with the verified schema hard-coded in
  `tools/validate.py:33-42`. All of it points inward at 8 plugins.
- **Proposal:** package the structural checks as a composite GitHub Action any
  plugin repo can `uses:`; green runs earn a listing in a `verified/` registry and
  a per-repo badge endpoint the window renders as a hall. Rides the normal line —
  pitch as an idea record first.
- **Why:** makes the foundry the trust layer for the whole Claude Code plugin
  ecosystem — a moat built from laws competitors would have to adopt wholesale.
  **Impact 5 · Effort 4.**

#### 14 · Ship the Foundry Network — NEW (advance existing records)
- **Evidence:** `foundry/network.json`, `foundry/records/foundry-network.md`
  (stage: spec) and `foundry/records/cross-foundry-exchange.md` (stage: idea)
  already sketch it, and `fork-a-foundry` is published — the replication vector
  exists; the federation doesn't.
- **Proposal:** a lightweight protocol: forks register via PR to `network.json`,
  each foundry's build emits a small interop feed, and the window cross-lists
  sibling shelves. Advance the existing records rather than opening new ones.
- **Why:** a self-replicating marketplace network has no precedent anywhere; it is
  the VISION's endgame and only this repo can start it. **Impact 5 · Effort 5.**
